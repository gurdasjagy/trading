#### ISSUE 1: Bybit `OrderLinkedID is duplicate` (Error 110072) - CRITICAL, Causes Legging Risk & Money Loss

**Root Cause:** In `rust_engine/src/bybit_gateway.rs`, the `BybitGateway` struct has a field `next_client_id: AtomicU64` initialized to `1`. This counter generates `orderLinkId` values like `rte0000000000000001`, `rte0000000000000002`, etc. The problem is that **this counter resets to 1 every time the engine restarts**. Bybit rejects orders with previously-used `orderLinkId` values (error 110072: "OrderLinkedID is duplicate") because Bybit remembers them for a period. Every single funding arb and stat arb order to Bybit fails with this error, causing:
- Funding arb: Short leg fills on Binance, long leg fails on Bybit, triggering emergency close (legging risk) and losing money on fees every ~30 seconds in a loop.
- Stat arb: Both legs fail repeatedly ("Both legs failed: Unknown error 110072").
- The engine burned through ~$1000+ in Binance fees from repeated open/emergency-close cycles during the log session.

**Exact Fix in `rust_engine/src/bybit_gateway.rs`:**
1. Change `next_client_id` initialization from `AtomicU64::new(1)` to use a **timestamp-based seed**: `AtomicU64::new(std::time::SystemTime::now().duration_since(std::time::UNIX_EPOCH).unwrap_or_default().as_millis() as u64)`. This ensures uniqueness across restarts.
2. Change the `next_order_link_id()` format to include a random component or use a UUID-like pattern: `format!("rte{:016x}", id)` is fine as long as `id` starts from a unique timestamp-based value.
3. Additionally, in `classify_bybit_error` in `rust_engine/src/execution_gateway.rs`, add error code `110072` to the match: `110072 => ExchangeError::Unknown { code: "110072".to_string(), message: ret_msg.to_string() }`. Currently 110072 falls through to the generic `Unknown` case, which is fine functionally but should be explicitly handled for clarity.

**Exact Fix in `rust_engine/src/binance_gateway.rs`:**
Apply the same timestamp-based seed fix to the Binance gateway's client order ID counter if it has the same pattern (check `next_client_id` initialization).

#### ISSUE 2: Dashboard Template Render Error - `Failed to render 'dashboard.html'`

**Root Cause:** In `rust_engine/src/dashboard_server.rs`, after a manual trade is opened, the `template_context()` method builds the Tera context. The `dashboard.html` template likely references variables related to the manual trade's position data (e.g., `positions` array entries with fields like `unrealized_pnl`, `entry_price`, `liquidation_price`, `stop_loss`, `take_profit`) that are either missing or have wrong types. The log shows:
```
[execution] Manual trade filled: buy BTC_USDT 0 contracts @ 0.0000 (order_id=)
[exit-eval] Tracking LONG position for sym 1 @ 0.0000 SL=0.0000 TP=0.0000
[lifecycle] Position netting: sym=1 added=0 net=0
[lifecycle] Tracking LONG 1 @ 0.0000 size=0 lev=10x notional=$0
```
The manual trade is recorded with **entry_price=0.0, size=0, order_id=""** because the Gate.io WS gateway returns `filled=0/0` and `id=""` in the ACK. The position JSON written to `DashboardState.positions_json` contains zero values, and when `dashboard.html` tries to render calculations (e.g., PnL percentage with division by zero entry price, or liquidation price computation), Tera throws a render error.

**Exact Fix in `rust_engine/src/gateio_gateway.rs`:**
1. The Gate.io WS order submission returns an initial ACK with `status=open, filled=0/0`. The execution thread must **wait for the fill confirmation** via the WS `futures.orders` subscription or poll via REST `GET /api/v4/futures/usdt/orders/{order_id}` before recording the position. Currently it records the position immediately from the ACK.
2. After `submit_order` returns the ACK, add a polling loop (up to 5 seconds, 200ms intervals) that calls `get_order_status(order_id, symbol)` to get the actual fill price and size. Only then send the `ManualPositionTrack` to the strategy thread.

**Exact Fix in `rust_engine/src/dashboard_server.rs`:**
1. In `template_context()`, when building the positions array for the template, filter out positions with `entry_price == 0.0` or `size == 0` (ghost positions from unfilled ACKs).
2. Add defensive template variables: ensure every position object in the JSON has `unrealized_pnl`, `liquidation_price`, `stop_loss`, `take_profit`, `mark_price`, `margin` fields (even if 0.0 or null) so the template never encounters undefined variables.

**Exact Fix in `crypto_trading_bot/templates/dashboard.html`:**
1. Add Tera `default` filters on all position fields that could be undefined: `{{ position.unrealized_pnl | default(value=0) }}`, `{{ position.liquidation_price | default(value="N/A") }}`, etc.
2. Guard division operations: `{% if position.entry_price > 0 %}` before computing PnL percentages.

#### ISSUE 3: No Real-Time Trade Updates on Dashboard After Manual Trade

**Root Cause:** The WebSocket broadcast in `spawn_ws_broadcast_task()` in `dashboard_server.rs` sends `positions` and `recent_trades` from `DashboardState`, but the execution thread never updates these after a manual trade fill. The manual trade flow is:
1. Dashboard POST `/api/manual-trade` -> sends to `manual_trade_tx` channel
2. Execution thread receives it, submits to Gate.io, gets ACK
3. Records position in `exit_evaluator` and `position_lifecycle` on the strategy thread
4. **But never calls `dashboard_state.set_positions_json()` with the updated position data**

The positions JSON in DashboardState stays as `"[]"` forever because no code path updates it with live position data from the exchange.

**Exact Fix in `rust_engine/src/main.rs` (execution thread):**
1. After a manual trade fill is confirmed, build a position JSON object with all fields: `symbol`, `side`, `entry_price`, `size`, `leverage`, `unrealized_pnl`, `liquidation_price`, `stop_loss`, `take_profit`, `mark_price`, `margin`, `pnl_pct`.
2. Call `dashboard_state.set_positions_json(serde_json::to_string(&positions).unwrap())` with the complete position data.
3. Also call `dashboard_state.set_trades_json(...)` to add the trade to the trades list.

**Exact Fix in the strategy/telemetry thread:**
1. In the telemetry loop (runs every 30s per logs), add a periodic position sync: query Gate.io REST `GET /api/v4/futures/usdt/positions` to get all open positions with real-time data (mark price, unrealized PnL, liquidation price, margin).
2. Format this data as JSON and call `dashboard_state.set_positions_json(...)`.
3. This ensures the WebSocket broadcast pushes real position data to the dashboard every 500ms.

**Exact Fix in `rust_engine/src/dashboard_server.rs` WebSocket broadcast:**
1. In `spawn_ws_broadcast_task()`, add position-specific fields to the broadcast payload: for each position, include `mark_price`, `unrealized_pnl`, `liquidation_price`, `stop_loss`, `take_profit`, `pnl_pct`, `margin`.
2. Add a `manual_trades` field to the broadcast so the frontend can distinguish manual from algo trades.

**Exact Fix in `crypto_trading_bot/templates/dashboard.html` and `crypto_trading_bot/static/js/main.js`:**
1. In the WebSocket `onmessage` handler in `main.js`, parse the `positions` array from the broadcast and update the trades section DOM in real-time.
2. For each position, render: symbol, side, entry price, mark price, unrealized PnL (with color: green/red), PnL %, liquidation price, SL, TP, margin, leverage.
3. Add a periodic DOM update (every 500ms matching the WS broadcast interval) that refreshes all position rows.

**Exact Fix in `rust_engine/src/multi_exchange/funding_arb_engine.rs`:**

2. After a legging risk event or both-legs-failed, insert the symbol with current time and increment failure count.
3. Before approving a new opportunity, check if the symbol is in cooldown: skip if `elapsed < 300 seconds * failure_count` (exponential backoff).
4. Clear the cooldown entry after a successful trade or after 30 minutes.
5. Add a maximum failure count (e.g., 3) after which the symbol is blacklisted for the session.
