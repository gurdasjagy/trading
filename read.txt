## 1. Root Cause: Gate.io Manual Trade Failure (`WS_REJECT: Not login`)

### Analysis
When you submit a manual trade via the dashboard, the Rust engine's `gateio_gateway.rs` constructs an order and sends it via the WebSocket API. The logs indicate that the API rejects the request with `Unknown error WS_REJECT: Not login`.

### Root Cause
Gate.io's WebSocket V4 API has a strict authentication sequence for private `event: "api"` requests (like `futures.order_place`). 
Currently, the codebase attempts to authenticate by embedding auth parameters into the `futures.orders` subscription message (`build_auth_subscribe_message`). While this successfully subscribes you to order updates, **it does not authenticate the WebSocket connection session** for active trading.

When you send a `futures.order_place` event, Gate.io requires the connection to have been explicitly authenticated using the dedicated `"login"` channel first. Because the engine skipped the global `login` event, Gate.io rejects the order placement.

### Recommended Fix
You must update `rust_engine/src/gateio_gateway.rs` to send a distinct `login` event immediately upon connection, before sending any subscriptions or API orders.
```json
{
  "time": <timestamp>,
  "channel": "login",
  "event": "login",
  "auth": {
    "method": "api_key",
    "KEY": "<YOUR_API_KEY>",
    "SIGN": "<HMAC_SHA512_SIGNATURE>"
  }
}
```
Wait for the `{ "event": "login", "result": { "status": "success" } }` acknowledgment before allowing the gateway to accept trades.

## 2. Institutional Gap Analysis: Funding Arbitrage Trading

The current setup detects cross-exchange funding spreads (e.g., Gate.io vs. Binance) and attempts to short the high-rate venue and long the low-rate venue. However, compared to top-tier institutional market-making bots, there are several critical gaps that will lead to real-money losses:

### Critical Gaps
1. **Legging Risk (Execution Asymmetry):** 
   Your bot fires two market orders simultaneously. If one exchange rejects the order (or has high latency) and the other fills, the bot is left with "naked" directional exposure (Delta != 0). Institutional bots use a "Maker-Taker" approach: they place a post-only limit order on the less liquid exchange first, and only when that *fills*, they aggressively hedge on the liquid exchange (e.g., Binance) using an IOC (Immediate-Or-Cancel) order.
2. **Precision and Contract Normalization (Size Basis):**
   Binance quotes in Base/Quote (e.g., 1 BTC), while Gate.io uses a quantum/contract multiplier (e.g., 1 contract = 0.0001 BTC). The current bot struggles to perfectly match the nominal sizes across venues. A tiny sizing mismatch results in residual delta exposure.
3. **Fee vs. Yield Math:**
   Funding rates are typically small (e.g., 0.01% to 0.05%). Taking liquidity (market orders) on both legs costs around `0.04% * 2 = 0.08%` in taker fees to enter, and another `0.08%` to exit. This `0.16%` friction completely wipes out the funding yield. Institutional bots strictly use limit orders (maker fees = 0% or negative rebate) to enter the spread.
4. **Funding Timestamp Misalignment:**
   Exchanges process funding at different times. If you hold a position across Binance (8h funding) and Bybit (could be continuous or staggered), you risk paying funding on one exchange before receiving it on the other. Institutional bots precisely time their entries seconds before the funding snapshot and exit immediately after.
5. **Cross-Venue Collateral Management:**
   If the market trends heavily in one direction, the short leg will accumulate massive unrealized losses while the long leg makes profit. Because the capital is isolated on two different exchanges, the short leg will get **liquidated** before you can manually transfer funds. There is no automated API withdrawal/deposit logic to auto-rebalance USDT between Binance and Gate.io to maintain margin health.

## 3. Root Cause: Multi-Exchange Leg Failures (Invalid Price & Insufficient Balance)

### Analysis
When the multi-exchange feature attempts a funding/stat-arb trade, the logs show: `bot legs failed for the order. short=invalid price and long= insufficient balance`.

### Root Cause: Invalid Price
The engine calculates a mid-price or crossing price (e.g., `71469.12345`) but sends it directly to the exchange API without snapping it to the exchange's specific `tick_size` (price precision). Gate.io and Binance will reject limit/stop orders if the price has too many decimal places or doesn't align with the tick size (e.g., step size of `0.5`). 

### Root Cause: Insufficient Balance
There are two reasons this happens in your architecture:
1. **Quantity Formatting:** The bot calculates position sizes in USDT (e.g., 50 USDT * 10x leverage = 500 USDT notional). If it sends `size=500` to Binance without dividing by the current BTC price, Binance interprets it as buying **500 BTC**, instantly triggering an "insufficient balance" error.
2. **Margin Monitor Scope:** In `rust_engine/src/main.rs`, the codebase previously instantiated separate instances of the margin monitor. The funding arb engine was checking a "local" margin state that had $0 recorded, while the actual API had the funds. Even with the recent `Arc<RwLock<>>` fix, if the balances aren't actively polled and cached *before* the arb check, the engine thinks it has 0 USDT. 

### Recommended Fixes
* **Tick/Step Size Normalization:** Implement a precision formatter. Before creating the `OrderIntent`, calculate `price = round(raw_price / tick_size) * tick_size` and `qty = round(raw_qty / step_size) * step_size`.
* **Standardize Sizes to Base Currency:** Ensure all leg calculations convert USDT nominal value into actual contract quantities (`USDT_notional / Price = Base_qty`), then convert that to the exchange-specific contract multiplier (e.g., Gate.io `quanto` multiplier).
